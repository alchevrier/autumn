# ADR 0017: Flyweight Zero-Allocation Model (Index Passing)

## Status
Proposed

## Context
In [ADR 0016](0016-pipelined-cache-affinity-scheduling.md), we established that the `@Pipelined` compiler plugin automatically generates hardware-sympathetic, structurally bounded arrays (Structure of Arrays / SoA) from standard Kotlin domain interfaces. 

However, a fundamental gap remained: **How does a developer interact with an object that physically does not exist?** 

If `OrderEvent` is erased by the compiler and physically stored as disparate `LongArray` and `IntArray` blocks, passing an `OrderEvent` as a parameter to a `@HotPath` function poses an architectural challenge. 
If we allocate an `OrderEvent` wrapper object on the heap simply to hold the array index and the getters/setters, we violate the strict zero-allocation boundaries required for L1 cache stability and re-introduce Java Garbage Collection pauses. 

We need a mechanism that provides standard Object-Oriented IDE ergonomics (methods, type-safety, interface boundaries) while remaining mathematically guaranteed to not allocate on the heap.

## Decision
We establish the **Flyweight Zero-Allocation Model utilizing Kotlin Inline Value Classes.**

### The Transformation Mechanism
When the Autumn K2 Compiler encounters a `@Pipelined` interface, it does two things:
1.  **Generates the Backing SoA:** It creates the primitive arrays (`IntArray`, `ByteArray`) to hold the data, sized to the requested capacity constraint.
2.  **Generates an Inline Value Class:** It emits an `@JvmInline` value class wrapper that implements the target interface. This wrapper contains only a single underlying primitive: `val index: Int`.

### 1. Developer Ergonomics (The Interface)
The developer continues to write heavily modeled domain code without caring about the underlying topographies:

```kotlin
@Pipelined(capacity = 64)
interface OrderEvent {
    var quantity: Int
    var side: Byte
}

@HotPath
fun processMarketData(order: OrderEvent) {
    order.quantity = 100 // Beautiful IDE autocomplete and type-safety
}
```

### 2. Physical AST Transformation (The Bytecode)
The Autumn compiler physically rewrites that code under the hood to completely eradicate the object. The `OrderEvent` reference is evaluated entirely as the primitive `Int` index generated by the inline flyweight.

The actual JVM Bytecode / Kotlin Native LLVM Bitcode executes this:
```kotlin
// The Array Backing (SoA)
val synthetic_OrderEvent_quantity = IntArray(64)
val synthetic_OrderEvent_side = ByteArray(64)

// The compiled Hot Path function
fun processMarketData(order_index: Int) {
    // order.quantity = 100 becomes a native vector array instruction
    synthetic_OrderEvent_quantity[order_index] = 100 
}
```

### The Scheduler Handoff
When the Cache-Affinity Scheduler parses an incoming packet from a `@BoundaryChannel`, it evaluates the hash/key and derives an array index (e.g., `0`). It instantiates the flyweight wrapper `val event = OrderEventFlyweight(0)` and hands it to the hot logic. 
Because `OrderEventFlyweight` is marked `inline`, the Kotlin compiler optimizes the instantiation away. No heap memory is ever requested from the OS allocator. 

## Consequences
*   **Positive:** Absolute zero-allocation (0-alloc) across the entire hot path, ensuring complete GC isolation and absolute L1 latency bounds.
*   **Positive:** Pristine developer experience. Teams can build complex domains using Domain-Driven Design (DDD) rules while the compiler forcibly translates them into Data-Oriented Design (DoD) hardware pipelines. 
*   **Negative:** Developers cannot use polymorphism or store these flyweight objects in standard Collections (`List<OrderEvent>`), as attempting to do so forces boxing (heap allocation). (Note: Standard collections are already blocked in HotPaths by the `AllocationFirChecker`).
