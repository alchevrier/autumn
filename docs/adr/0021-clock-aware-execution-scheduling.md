# ADR-0021: Clock-Aware Execution Scheduling

## Status
Accepted

## Context
Autumn has successfully demonstrated that compiling Kotlin abstractions into lock-free primitive sequences drops cross-core pipeline handoffs to hardware limits (~29ns). Currently, the K2 compiler generates reactive FSM blocks (`while(true) { process() }`). 

While optimal for generic JVM testing, true "Circuit-Based Programming" on bare metal mimics FPGA/ASIC execution where systems are deterministic, acting not reactively to unpredictable events, but strictly synchronously to physical hardware clock signals. The reactive model makes cache behavior unpredictable under micro-bursts and removes our ability to mathematically prove execution correctness at compile-time.

## Decision
We will transition the execution boundaries generated by the `TopologySynthesisTransformer` from reactive `while(true)` loops to **Clock-Aware FSM Tick Scheduling**.

1. **Elimination of Busy-Spinning:** Handlers will no longer poll rings directly in infinite `while(true)` spinloops. Busy-spinning pegs CPU cores at 100%, fighting the OS scheduler and causing severe thermal throttling on mobile devices. Instead, the compiler synthesizes topological bindings into an `AutumnScheduler`/`Arbiter` combination.
2. **Self-Scheduling Synthetic Clocks:** Developers can attach relative execution `@Frequency` boundaries to their pipelines. Rather than burning clock cycles repeatedly querying physical timers ("is it time yet?"), the framework calculates the next tick and relies on platform-idiomatic mechanisms to *self-schedule* and wake up cooperatively:
    - **JVM:** High-resolution OS timers (e.g., `LockSupport.parkNanos()` or `ScheduledExecutorService`).
    - **Android/Web:** Hardware display bounds scheduling themselves for the next VSYNC (e.g., Android `Choreographer.postFrameCallback` or Web `requestAnimationFrame`).
    - **Linux Native:** POSIX high-res timers (`timerfd_create`) mapping directly into the OS event loop.
3. **Drift Compensation & Assumed Time:** System calls to fetch time natively (`System.nanoTime()`) are expensive and stall processor execution ports. Therefore, the scheduler limits OS time queries. Once bounded, it operates on execution *assumptions*—it schedules future progression steps logically and assumes the physical clock holds true until periodic boundary checks trigger drift compensation protocols to re-sync reality.
4. **Batch Coalescing:** By executing strictly against these synthetic hardware clocks, incoming network payload bursts (e.g., AF_XDP DMA transfers or standard JSON web-socket streams) naturally aggregate/batch between execution cycles without software intervention. The synthetic tick wakes once, executing the entire sequence linearly over L1 cache cleanly.

## Consequences
- **Positive:** Cache prediction and memory layout utilization become 100% deterministic under heavy burst contention.
- **Positive:** We natively decouple execution timing across platforms (HFT hardware uses microsecond CPU limits; Android naturally aligns strictly on 16ms Display frame clock pulses).
- **Positive:** We can achieve the "holy grail" of static software proofs — calculating cycle limits dynamically matching the instruction-level topology footprint and warning the developer if their pipeline logic exceeds the physical budget between clock ticks natively in the IDE.
- **Negative:** The backend K2 compiler generation rules must be refactored significantly to wire functions tightly through an `AutumnScheduler.tick()` structure instead of generating raw bytecode loops dynamically.
